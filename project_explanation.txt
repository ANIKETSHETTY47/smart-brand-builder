================================================================================
SMART BRAND & DOMAIN AVAILABILITY PLATFORM
Comprehensive Project Architecture & Infrastructure Guide
================================================================================

1. PROJECT OVERVIEW & OBJECTIVE
--------------------------------------------------------------------------------
The "Smart Brand & Domain Availability Platform" is a scalable, cloud-native
web application designed to generate brand identities for users based on their
business name and industry category. 

It accomplishes three major tasks:
1. It intelligently generates domain name variations based on naming algorithms.
2. It assigns brand attributes (aesthetic style, icon shape, and tone) using
   deterministic rule-based logic (no AI/ML).
3. It fetches a dynamically generated SVG logo from a peer's external API.

The entire system is deployed on AWS (Amazon Web Services) utilizing a 100%
Serverless architecture. This guarantees infinite scalability, zero server 
maintenance, and extreme cost efficiency (you only pay per request).


2. THE THREE REQUIRED APIs
--------------------------------------------------------------------------------
This project successfully satisfies the architectural requirement of consuming
and integrating three distinct APIs:

A. "My API" (The Custom Backend API)
- This is the core API developed for this specific project.
- It is hosted on AWS API Gateway and powered by an AWS Lambda function.
- It acts as the central orchestrator. It receives data from the frontend, applies
  domain generation algorithms and branding logic, and orchestrates calls to the 
  other two APIs.
- Endpoint: `POST /api/brand-domain/`

B. "Peer API" (Classmate's Logo Generator API)
- This is an external API written by a classmate.
- "My API" consumes this service securely over the internet.
- It accepts a company name, style, and icon type, and returns an SVG logo.
- Fallback Resilience: If the Peer API goes offline, times out, or returns a 500 error, 
  "My API" gracefully catches the error and generates an internal placeholder 
  SVG locally using strict Python dimensions, ensuring the frontend never crashes.

C. "Public API" (WhoisXML Domain Checker API)
- This is a publicly available third-party web service.
- "My API" consumes this to check if a specific domain is registered or available.
- Security: The secret API Key is never hardcoded. It is pulled safely at runtime
  from the AWS Systems Manager Parameter Store (SSM).


3. CLOUD ARCHITECTURE & AWS SERVICES USED
--------------------------------------------------------------------------------
This project utilizes modern FaaS (Function-as-a-Service) principles.

A. AWS S3 (Simple Storage Service)
- Hosts the entire Frontend web application (index.html) infinitely and securely.
- Secondary S3 bucket acts as an archival store to save the returned SVG logos.

B. AWS API Gateway
- Acts as the "front door" for the backend. It safely receives HTTP POST requests
  from the user's browser, handles proxy integration, and routes the traffic quickly
  to AWS Lambda.

C. AWS Lambda (FaaS Computation)
- The brain of the API. Lambda runs the Python backend only when activated by
  API Gateway or an SQS message. It scales automatically from 1 request to 10,000 
  requests per second without provisioning a single server (unlike Elastic Beanstalk).

D. AWS SQS (Simple Queue Service)
- Used for asynchronous scalability. When a user submits a request, Lambda handles
  it, but delegates the "Audit Logging" by firing a quick message to SQS. SQS safely
  holds the message until a background worker processes it, ensuring the user gets 
  their response back instantly without waiting for the database log to finish.

E. AWS DynamoDB
- Serverless NoSQL Database. Used for two separate, highly scalable tables:
  1. `brand-domain-results`: Stores the history of generated domains and brand values.
  2. `brand-domain-audit`: Stores the timestamps and logging data processed from SQS.


4. HIGH-LEVEL DATA FLOW (STEP-BY-STEP)
--------------------------------------------------------------------------------
Phase 1: Generation Flow
1. User loads the Frontend website from the AWS S3 Bucket URL.
2. User types a "Business Name" (e.g., Nexus) and selects a "Category" (e.g., Tech).
3. The frontend JS `fetch()` method executes an HTTP POST to the API Gateway.
4. API Gateway invokes the `lambda_main.py` script.
5. The Lambda script cleans the strings and generates 4 different domain variations
   (e.g., nexus.com, getnexus.com, nexusapp.com, nexus.io).
6. The Lambda script checks the category against strict Rule-Based logic arrays
   to determine styling values (e.g., Tech = hexagon icon, blue color).
7. The Lambda calls the "Peer API" to get the SVG Logo utilizing these exact attributes.
8. The Lambda instantly dumps an archival metric into DynamoDB and the Logo S3 bucket.
9. The Lambda fires an async message to the SQS queue indicating a 'SUCCESS'.
10. The Lambda finishes its run by sending a JSON package full of DOMs and SVGs 
    back to the user's browser, completing the "My API" flow.

Phase 2: Independent Asynchronous Audit Flow
1. The SQS Queue receives the 'SUCCESS' message from Phase 1.
2. SQS safely invokes a second, totally isolated Lambda (`lambda_audit.py`).
3. This audit Lambda timestamps the payload and writes it to the secondary DynamoDB
   audit table, heavily decoupling our architecture.

Phase 3: Deep Verification Flow (Public API)
1. The user visually reviews the 4 generated domains on the frontend.
2. The user clicks "Check Availability" on a specific domain (e.g., nexus.com).
3. A second, isolated HTTP POST hits `lambda_main.py` with `action: check_domain`.
4. `lambda_main.py` detects this action and routes it to `domain_checker.py`.
5. The code securely un-encrypts the API Key from AWS SSM.
6. The code queries the Public "WhoisXML API" and gets definitive registration status.
7. The API immediately returns "Available" or "Taken", painting the frontend green/red.


5. FOLDER & FILE STRUCTURE EXPLAINED
--------------------------------------------------------------------------------
The project is split functionally between Infrastructure, CI/CD, Frontend, and Backend.

► Root Directory 
- `aws_setup.sh`: A 150-line infrastructural bash script. When run in AWS Learner
   Lab, it automatically creates every Cloud component (S3, Dynamo, IAM Roles, SQS, 
   API Gateway) without clicking buttons in the AWS UI. This is called Infrastructure 
   as Code (IaC).
- `architecture_and_report.md`: Contains text-diagrams and scalability justifications.
- `peer_api_documentation.md`: Contains the precise cURL payloads your classmates 
   can use to target your API over the internet.
- `README.txt`: Project submission index and local testing notes.

► `.github/workflows/deploy.yml` (CI/CD Pipeline)
- This is the Continuous Integration / Continuous Deployment engine.
- Every time code is 'Pushed' to the main GitHub branch, GitHub servers read this 
  YAML file and automatically run Python Pytests, zip the newest backend codebase 
  up, and overwrite the live AWS Lambda functions seamlessly with zero downtime. 

► `tests/`
- Contains `test_brand_logic.py`, `test_domain_logic.py`, `test_fallback_svg.py`.
- These are Automated Unit Tests using the `pytest` framework to ensure that string 
  cleaning, array scoring, and XML SVG fallback mapping logic executes perfectly 
  before deploying to production.

► `frontend/`
- Contains `index.html`. This is a Single Page Application (SPA).
- We entirely bypassed massive bloated frameworks like React to favor vanilla
  Javascript, raw HTML5, and native CSS Variables ensuring lightning-fast static 
  delivery from the AWS S3 bucket. It perfectly catches API Gateway network 
  failures and renders spinning loaders for fluid UX.

► `backend/` (The Core Engine)
This directory is zipped up and sent directly into the AWS Lambda runtime.

- `lambda_main.py`: The "App.js" of the system. The central traffic cop that 
   receives the inbound JSON from API Gateway, handles parsing, determines if the 
   user clicked 'Generate' or 'Check', and delegates work to the helper modules below.
- `lambda_audit.py`: The isolated background worker invoked ONLY by SQS to handle logs.

- `domain_logic.py`: The algorithm that builds domain variations and dynamically 
   scores them based on string length and hyphen penalties.
- `brand_logic.py`: The dictionary mapping tables taking "food" or "finance" strings 
   and mapping them deterministically to specific shapes and tones (no AI logic).

- `domain_checker.py`: The secure client connection out to the WhoisXML Public API.
- `logo_client.py`: The secure client connection out to the Classmate Peer API.
   Contains the robust `get_fallback_svg()` dynamic generator to prevent outages.

- `sqs_publisher.py`: Boto3 helper for writing non-blocking payloads to the SQS queue.
- `storage.py`: Boto3 helper abstracting DynamoDB table insertions and S3 bucket 
   image artifact insertions.
